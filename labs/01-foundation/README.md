# Session 01 Lab Guide: Foundations of Data Engineering & Apache Airflow 3.x

ยินดีต้อนรับเข้าสู่เซสชันแรกของหลักสูตร ในเอกสารฉบับนี้เราจะพาคุณไปเรียนรู้บทบาทของ Data Engineer แนวคิดสากลในการจัดการระบบท่อส่งข้อมูล (Data Pipeline) และทำความรู้จักกับ Apache Airflow 3.x

---

## 1. บทบาทของ Data Engineer (DE คือใคร?)

**Data Engineer (วิศวกรข้อมูล)** มีหน้าที่หลักในการสร้างและบำรุงรักษา "โครงสร้างพื้นฐาน" (Infrastructure) และ "ระบบท่อส่งข้อมูล" (Data Pipelines) เพื่อเชื่อมต่อข้อมูลจากแหล่งต้นทางต่างๆ (เช่น เว็บไซต์, แอปพลิเคชัน, เซ็นเซอร์, ฐานข้อมูลภายใน) นำไปผ่านกระบวนการคัดกรอง ทำความสะอาด จัดรูปแบบ แล้วโหลดเข้าสู่แหล่งเก็บข้อมูลปลายทางส่วนกลาง (เช่น Data Lake หรือ Data Warehouse) เพื่อนำไปวิเคราะห์ต่อ

### ความแตกต่างระหว่าง ETL และ ELT

| คุณลักษณะ | ETL (Extract-Transform-Load) | ELT (Extract-Load-Transform) |
| :--- | :--- | :--- |
| **กระบวนการ** | สกัด -> แปลงรูปแบบที่เซิร์ฟเวอร์ท่อพักข้อมูล -> โหลดเข้าฐานข้อมูลปลายทาง | สกัด -> โหลดเข้าฐานข้อมูลดิบตรงๆ -> แปลงรูปแบบข้างในฐานข้อมูลโดยตรง |
| **ความเหมาะสม** | เหมาะกับข้อมูลขนาดเล็กถึงปานกลาง ที่ต้องการควบคุมความสะอาดก่อนจัดเก็บ | เหมาะกับระบบ Big Data สมัยใหม่ (เช่น Cloud Data Warehouses) ที่การประมวลผลปลายทางทำได้เร็วมาก |
| **ข้อดี** | ข้อมูลปลายทางจะสะอาดและประหยัดพื้นที่เก็บข้อมูลทันที | ทำงานสกัดและจัดส่งได้รวดเร็วกว่า ข้อมูลดิบไม่ตกหล่นระหว่างแปลงรูปแบบ |

## 2. ทำความรู้จักกับ Apache Airflow 3.x

**Apache Airflow** คือระบบเปิด (Open-source platform) สำหรับกำหนดรูปแบบ ทำการจัดเวลา และตรวจสอบตรวจสอบข้อมูลในระบบท่อส่งข้อมูลด้วยโปรแกรมภาษา Python โดยในเวอร์ชัน **Airflow 3.x** นี้ มีการปรับปรุงโครงสร้างพื้นฐานให้มีความทันสมัย รองรับการทำ AI Integration และมีความปลอดภัยสูง

### ส่วนประกอบหลักบนหน้าจอควบคุม (Airflow UI)

เมื่อรันระบบ Airflow ขึ้นมาและเข้าไปทางบราวเซอร์ จะพบแถบควบคุมที่สำคัญดังนี้:

1.  **DAGs (Directed Acyclic Graphs)**: หน้าแรกแสดงรายการท่อส่งข้อมูลทั้งหมด ใน Airflow แต่ละโฟลว์จะเรียกว่า "DAG" ซึ่งเป็นชุดงานที่เชื่อมโยงกันแบบระบุทิศทางโดยไม่มีการวนลูปซ้ำแบบไม่สิ้นสุด
2.  **Grid View (มุมมองตาราง)**: มุมมองหลักในการเฝ้าสังเกตสถานะการรันของ Task ในแต่ละรอบ ทำให้เห็นภาพรวมได้ชัดเจนว่า Task ไหนรันผ่าน (สีเขียว) หรือล้มเหลว (สีแดง)
3.  **Logs (หน้าประวัติระบบ)**: เมื่อ Task ทำงานล้มเหลว เราสามารถกดเข้าไปตรวจสอบรายละเอียดข้อผิดพลาด (Error messages) หรือข้อมูล Debug ในหัวข้อ Logs ได้ทันที ซึ่งสำคัญมากในการทำ Troubleshooting
4.  **Admin -> Connections**: ส่วนสำหรับตั้งค่า Credential, Username/Password หรือ API Keys สำหรับต่อเชื่อมบริการภายนอก เช่น ฐานข้อมูลเวกเตอร์ หรือ Google AI Studio (Gemini) โดยโค้ด DAG จะเรียกใช้ผ่าน Connection ID ทำให้ไม่มีรหัสผ่านความปลอดภัยรั่วไหลลงไปในตัวโค้ดหลัก

### แนวคิดหลักในการออกแบบ DAG ที่ดี
*   **Atomic Tasks**: แต่ละ Task ใน DAG ควรทำงานเพียงเรื่องเดียว (Single Responsibility) เช่น Task 1 ดึงไฟล์, Task 2 แบ่งข้อความ, Task 3 เขียนเข้า Vector Database เพื่อให้ง่ายต่อการแก้ไขและรันเฉพาะส่วนที่พังใหม่
*   **Idempotency**: การสั่งรัน Task เดิมซ้ำๆ ในวันเดิม จะต้องได้ผลลัพธ์เหมือนเดิมทุกประการ ไม่มีการเขียนข้อมูลซ้ำเบิ้ล ซึ่งทำได้โดยการใช้คำสั่ง "ลบ/แทนที่" (Overwrite) แทนการเขียนต่อท้าย (Append) เสมอ
